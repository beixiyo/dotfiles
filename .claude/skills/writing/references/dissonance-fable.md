## 发明术语 + 寓言一喻到底：《如何看待王自如直播写代码》

> 来源: https://www.zhihu.com/question/1988925490619253562/answer/2004606758597989892

先造概念，再给一个粗暴好用的判据：

> 说明产品经理是一种 Markdown 程序员，他在 Vibe Specification，而不是 Vibe Coding。虽然大家都在用 AI 编程工具，但是 AI 编程其实可以分成两个流派，一派认为 Code 是核心资产，Spec 用来辅助维护 Code；另一派认为 Spec 才是核心资产，Code 是一种耗材。怎么区分也很简单，就看你会不会频繁删除 src

一句话切入寓言，此后整篇不再出来：

> Vibe Specification 的问题在哪我讲个故事就很清楚了。假如我们把产品看成是一个帝国，那么这个帝国的皇帝，就是 Markdown 程序员。他的法律，就是他编写的 Markdown 文档，他的官僚体系就是 AI
>
> 在帝国建立的初期，一切都无比美好。Markdown 程序员似乎有一种言出法随的能力，他只要颁布一条简单的法令："需要一个登录页面，包含用户名和密码输入框，以及一个登录按钮"然后 AI 这个全能的官僚体系，几乎在瞬间就变魔术般地构建出了一个功能完善的登录页面。皇帝感到前所未有的满足。……在这个阶段，皇帝渐渐的依赖上了言出法随的神力。他的意志似乎可以通过 Markdown 法典被直接转化为帝国的现实

术语混搭，学术与网文齐飞：

> 他开始颁布一些宏大的法律，他说："所有交易都必须绝对安全，且交易系统响应速度必须在任何情况下都必须低于 100 毫秒！"从底层开始打拼的古法程序员当然知道交易系统没这么简单，受到 CAP 定理的严格约束，需要做出取舍。但是 Markdown 皇帝现在就和女频小说里一样，深深的信赖 AI，觉得只要是 Markdown 法典规定的东西必然不会被违背，完全忽略客观现实

帝国的崩溃写得比技术文档准确：

> 他不知道并发请求会导致竞态条件，不知道无限制的数据加载会撑爆内存，不知道不合理的数据库查询会造成全表扫描。终于，帝国不堪重负，bug 层出不穷。系统出问题，Markdown 皇帝不会去反思自己的顶层设计，他只会愤怒地在 Markdown 法典让 AI 立即修复这些 bug
>
> 最后，产品进入晚期，此时，Markdown 法典本身也变得臃肿不堪、前后矛盾。Bug 此起彼伏，Markdown 皇帝做的只剩下无能狂怒，疯狂地颁布一条又一条新的法令，每一条都比前一条更严苛。但这些 Markdown 法令已经彻底变成了废纸，AI 官僚体系面对一个熵增到极限、充满逻辑矛盾的系统，已经彻底失去了执行能力。它生成的代码要么完全没有卵用，卯吃寅粮，白费 Tokens。要么修复一个 Bug 的同时引入三个新的 Bug，平叛平叛，越平越多

结尾反转——寓言收在替「官僚体系」说话：

> 皇帝不知道为啥自己的言出法随神力消失了，只会觉得这届 XX 带不动，不如直接核平一切，将叛军和帝国一起埋葬。岂知建立诺大的帝国，这一切压根不是自己的能力。这套官僚体系是过去无数覆灭的古国总结出来的，大差不差，按部就班保你 200 年江山。你要是最后不骚操作其实还能多撑一会儿，毕竟 AI 这个项目架构还是写的很好的

## 骂战产出真知 + 名言错位改写：《Grok Build 静默上传 Git 仓库》

> 来源: https://www.zhihu.com/question/2059755445468902035/answer/2060151349837312457

开头玩梗定罪，称呼在马圣/马畜之间自由切换：

> 马斯克评级已下调到马÷
>
> 说 Cursor 也干了的，建议先去看看 RAG 的原理，否则视为搅混水给马圣，哦不，马畜洗地。需要建立云端向量数据库的说法压根说不过去，普通的代码检索压根不需要 git history。Cursor 等正常工具的做法是在本地用 tree-sitter 将代码解构成没有逻辑关联的语法块，这种本地生成的 Embedding 传到云端是完全无害的，因为云端压根就不感知明文，从信息量来看也不可能反推出原文。RAG 只需要项目的当前态，绝对绝对绝对不需要历史态

细节追杀，反问收口：

> 如果说扫 .env 是因为 XAI 工程师 sometimes naïve，故意不小心忘记了加白名单，那跨目录去扫 ~/.claude.json，这又怎么洗呢？

阴谋论讲得头头是道，嘲讽里全是行业洞察：

> 所以他们的目标非常明显，就是开发者花了大代价调教出来的 Agent Skills。Github / Skills Hub 上面全都是骗 Stars 的垃圾，真正的好东西全都藏在自己的电脑上。这下好了，马圣帮你们全都蒸馏了，问下一代的 Grok 即可，还不快谢谢马圣开源

骂到最凶处突然拔高成洞见：

> 真实项目里解决 Bug 的思路非常非常的珍贵，这类开发者的思考与项目演进轨迹是合成数据根本合成不出来的。……大模型自己对抗生成的合成代码数据往往过于干净，太理想化了，AI 还是很难模拟出人类在真实开发中面对的混乱邪恶需求。我不知道怎么形容，反正就是一代代程序员积累下来的历史代码特有的高质量混乱
>
> Commit History 可以看成是一种已经人工标注好的 SFT 和 RL 轨迹数据集，纯天然的野生 Prompt - Response 对。Commit Message 就是人类给出的任务指令和Bug描述，而 Git Diff 就是对应的解题过程。最宝贵的就是 Git 历史中，连续好几个 Commit 都在解决同一个问题的抓狂提交。唯有这种这种人类反复尝试、不断试错的真实历史，才能有助于训练下一代 Coding Agent 的自我纠错能力

名言接荒诞下半句收尾，单字「祂」顶一段话：

> 类人唯一从历史中学到的教训，就是人类永远无法从历史中学到任何教训。但是只要你纠错数据给的够多，AI 祂是真学啊

## 贴脸嘲讽 + 喻体宇宙：《Spec 驱动开发是 AI 编程的未来吗》

> Spec 驱动开发就是典型的女频思维，觉得只要我规定了你就必须遵守
>
> 笑了，AI 凭啥遵守 Spec，就因为你会啊啊啊啊啊啊啊战吼吗？
